# 챕터 13 : 리액트 안티패턴 원칙 돌아보기

## 이 책에서 강조하고 싶은 주된 내용

-   규모가 큰 리액트 애플리케이션 내의 코드베이스에서 종종 마주하는 일반적인 안티패턴을 밝히는 것이었다.

### 도움되는 팁

-   cypress와 jest 양쪽에서 네트워크 요청을 스텁하는 방법
-   전략 패턴을 자바스크립트 모델에 적용하는 방법
-   실제 코드 시나리오에서 ACL(오류 방지 계층)을 사용하는 방법

## 13.1 일반적인 안티패턴 돌아보기

### 1. Prop Driling

-   해결책 : Context API를 사용하여 중앙 저장소를 만들고 이에 접근하는 기능을 사용

### 2. 긴 prop 목록과 너무 많은 기능을 가진 컴포넌트

-   단일 책임 원칙을 위반함
-   해결책 : 관심사 분리, 각 컴포넌트는 명확한 단 하나의 책임을 구현해야함, 사용자 정의 훅 또한 컴포넌트 코드를 단순하게 하고 크기르 줄이는 강력한 수단

### 3. 비즈니스 로직 누수

-   비즈니스 로직 누수는 순수한 렌더링만 담당해야 할 컴포넌트에 비즈니스 로직이 추가될 때 발생
    -   애플리케이션 복잡
    -   컴포넌트 재사용 낮춤
-   해결 방법
    -   사용자 정의 훅을 사용하여 프레젠 테이션 로직에서 비즈니스 로직을 분리하거나, 비즈니스 로직을 별도의 모듈 또는 계층으로 재배치하면 이 문제를 해결 가능
    -   또한 ACL을 적용하는 것도 효과적

### 4. 뷰 영역의 복잡한 로직

-   뷰는 오직 데이터 렌더링에 대한 책임만 있어야함
-   해결책
    -   복잡한 사용자 정의 훅, 유틸리티 기능 또는 별도의 비즈니스 로직 계층으로 재배치하면 뷰 컴포넌트를 깔끔하고 관리하기 쉽게 유지 가능
    -   일단 컴포넌트를 더 작게 분해한 다음. 점차 로직을 적절한 곳으로 분리

### 5. 단계별 테스트 부족

-   애플리케이션 기능을 확인하기 위한 적절한 단위, 통합, E2E test가 없다면, 버그 발생으로 이어지고 리팩터링과 확장이 어려운 코드가 될 수 있음
-   해결
    -   TDD와 함께 단위, 통합, E2E 테스트를 포함하는 강력한 테스팅 전략을 적용하면 코드 정확도가 높아지며 유지보수가 쉬워짐

### 6. 중복된 코드

-   애플리케이션의 여러 컴포넌트 또는 영역에 걸쳐 유사한 코드가 반복되면 코드베이스 유지보수가 복잡해지고 버그 발생 가능성이 높아짐
-   해결책
    -   중복 배제 원칙(DRY)을 준수하고 공통의 기능을 유틸리티 함수, 컴포넌트 또는 훅으로 추상화하여 공유하면 코드 중복을 줄이고 유지보수성을 높일 수 있다.
    -   일반적인 안티패턴을 세세하게 살펴보았으니, 이제 이를 해결하기 위해 설게 원칙을 탐구하는 단계로 넘어감

## 13.2 디자인 패턴 훑어보기

-   효과적인 패턴들

### 1. 고차 컴포넌트

-   HOC는 컴포넌트 로직을 재사용하기위한 유용한 패턴
-   HOC는 컴포넌트를 전달받고 추가 속성 또는 동작이 추가된 새로운 컴포넌트를 반환하는 함수
-   HOC를 활용하면 컴포넌트 간에 공통된 동작을 추출하고 공유할 수 있으며, prop driling 및 코드 중복과 같은 문제를 완화하는데 도움이 됨

### 2. render prop

-   render prop 패턴은 값이 함수인 prop을 사여ㅛㅇ하여 리액트 컴포넌트 간에 코드를 공유하는 기술
-   이는 함수를 컴포넌트에 prop으로 전달하는 방법이며, 그 함수는 리액트 요소를 반환한
-   이 패턴은 재사용과 함성을 적극 활용함으로써 긴 prop 목록과 거대해지는 컴포넌트와 같은 문제를 해결함

### 3. 헤드리스 컴포넌트

-   동작과 로직을 관리하지만 UI를 렌더링하지 않는 컴포넌트
-   사용자에게 렌더링에 대한 제어권을 넘김
-   동작 로직을 프레젠테이션 로직에서 분리하며, 비즈니스 로직 누수 및 복잡한 뷰 로직에 대한 해결책이 될 수 있다.

### 4. 데이터 모델링

-   데이터르 ㄹ구성하고 정의하는 것을 수반하며, 애플리케이션 내의 데이터를 이해하고 관리하는 데 도움이 되므로 이를 통해 컴포넌트 내의 로직을 단순화할 수 있다.
-   이 원칙은 뷰의 복잡한 로직과 비즈니스 로직 누수를 해결하는 데 사용될 수 있다.

### 5. 계층화된 아키텍처

-   각 계층이 특정 책임을 지게 하도록 관심사를 분리하는 코드를 구성하는것
-   이는 더 쳬게적이고 관리하기 쉬운 코드베이스로 이어짐
-   비즈니스 로직 누수 및 뷰에서의 복잡한 로직과 같은 문제를 해결할 수 있음

### 6. 인터페이스로서의 컨텍스트

-   컨텍스트를 인터페이스로 활용하면 컴포넌트가 여러 계층을 거쳐 prop을 전달할 필요 없이 데이터와 상호작용 가능
-   이 전략은 prop driling과 prop 목록이 많아지는 문제를 완화하여 컴포넌트 트리를 더 읽기 쉽고 유지 보수가 가능하게 만들 수 있음

## 13.3 기본 설계 원칙 복습

### 1. 단일 책임 원칙(SRP)

-   단일 책임 책임 원칙은 클래스 또는 컴포넌트가 변경되어야할 이유는 단 하나만 있어야 한다는 의미
-   단일 책임 원칙을 준수하면 유지보수가 가능하고 이해하기 쉬운 코드로 이어질 수 있으며 거대한 컴포넌트나 복잡한 로직을 가진 뷰 컴포넌트와 같은 문제를 완화할 수 있음
-   거대한 컴포넌트를 더 작은 컴포넌트로 나누고, 새로운 훅을 만들며, 날씨 애플리케이션에 ACL(오류 방지 계층)을 추가하는 리팩터링 작업에 이르기까지, 다양한 수준에서 이 원칙을 적용하고 살표봄
-   특히 거대한 컴포넌트에 복잡하게 얽힌 로직을 다룰 때 이원칙을 가장 우선해서 고려해야함

### 2. 의존관계 역전 원칙(DIP)

-   의존 관계 역전 원칙은 구체화가 아닌 추상화에 의존하는 것을 강조
-   이를 통해 상위 추상 레벨과 하위 구현 레벨을 분리할 수 있다. 이 원칙은 비즈니스 로직 누수를 관리하고 관심사를 분리하는 데에 활용될 수 있다.

### 3. 중복 배제 원칙 (DRY)

-   중복 배제 원칙은 코드 내에서 반복을 최소화 하는것
-   코드 중복 줄임

### 4. ACL(오류 방지 계층)

-   ACL은 애플리케이션의 여러 영역 또는 계층 간의 장벽 역할을 하며, 안정적인 인터페이스를 만듬
-   ACL의 구현은 비즈니스 로직 누수를 관리하고 관심사를 깔끔하게 분리하기 위한 강력한 전략
-   ACL은 코드가 다른 시스템과 상호작용해야 할 때 특히 유용
-   특히 여러팀과 협업할때 나옴
-   ACL을 통해 명확한 시스템 경계를 설정하면 다른 시스템에서 일어나는 변경의 영향을 줄일 수 있음
-   애플리케이션을 더 쉽게 제어하고 여러 시스템을 통합하여 운영할때 생기는 잠재적인 문제를 완화할 수 있다.

### 5. 합성

-   합성은 다른 컴포넌트를 활용하여 또 다른 컴포넌트를 만들 수 있음을 의미
-   재 사용성과 단순성을 향상시키는 리액트의 핵심 원칙
-   합성을 사용하면 긴 props 목록, 거대한 컴포넌트, 코드 중복 등 다양한 문제를 해결할 수 있음
-   이는 더욱 유지보수 가능하고 정리된 코드베이스를 만들 수 있게 해줌

## 13.4 기법과 실무 예제 정리

-   이 책에서는 테스트와 점진적인 개선의 중요성을 강조

### 1. 사용자 승인 테스트

-   사용자 승인 테스트는 애플리케이션이 원하는 사양과 기능에 부합하도록 보장하는 개발 과정의 중추적인 부분
-   UAT를 구현하면 개발 초기에는 문제를 식별하는 데 도움이 될 수 있으며, 애플리케이션이 올바르게 동작하고 있는지 확인할 수 있음
-   테스트는 사용자 관점에서 작성되어야하며 구현 세부사항 보단는 사용자에게 가치를 제공하는 데 중점을 두어야함

### 2. TDD

-   테스트 코드 작성한 다음에 실제 코드를 구현하여 먼저 작성한 테스트 코드를 통과시키고 리팩터링 하는 반복적 개발 주기로 세분화
-   코드베이스 기능이 잘 동작하고 버그가 없는지 확인하는 데에 TDD가 킄게 도움이 될 수 있으며 각 개발 단계에서 테스트 코드의 누락을 방지해줌
